0:00-0:30 - Why Every Company Needs an IDP

The data is clear: seventy-three percent of developers say onboarding to a new service takes too long. Sixty-one percent can't find the documentation they need. Fifty-four percent have deployed incorrectly at least once because the process was unclear. An IDP solves all three by providing self-service infrastructure, golden path templates, and integrated documentation. But building an IDP traditionally takes twelve to eighteen months and a dedicated platform team of six to ten engineers. 0:30-1:00 - What an IDP Actually Includes
Visual: IDP component diagram: Service catalog, Golden path templates, Self-service provisioning, CI/CD integration, Observability dashboards, Documentation portal.
Voiceover: An Internal Developer Platform isn't a single tool — it's an integrated experience. Service catalog: a searchable registry of every service, API, and library in your organization with ownership, status, and dependencies. Golden path templates: pre-approved, production-ready templates for creating new services — click a button, get a repo with CI/CD, monitoring, and security configured. Self-service provisioning: developers request databases, queues, and cloud resources through a portal, not tickets. CI/CD integration: every service created through the IDP gets a pipeline automatically. Observability dashboards: pre-built monitoring for every service type. Documentation portal: docs that stay current because they're generated from the platform metadata, not written manually.

1:00-1:30 - AI-Accelerated Platform Building
Visual: AI agent building IDP components: analyzing existing services, generating templates, creating documentation, configuring pipelines. Speed comparison: 18 months → 7 months.
Voiceover: AI agents accelerate IDP construction in four ways. Service catalog population: the agent scans your Git repositories, CI/CD systems, and cloud accounts to automatically catalog every service with metadata — language, owner, dependencies, deployment target. Golden path generation: analyze your best-maintained services, extract common patterns, and generate templates that encode those patterns. Documentation generation: create and maintain service documentation from code analysis, API schemas, and deployment configurations — always current, never stale. Pipeline configuration: generate CI/CD pipelines tailored to each service type following your organization's standards. Teams using AI agents report seven-month IDP build times versus eighteen months manual.

1:30-2:00 - Golden Paths: The Core Value
Visual: Golden path flow: Developer clicks "Create Service" → Selects template → IDP generates repo, CI/CD, monitoring, docs → Developer codes. 45 minutes total.
Voiceover: Golden paths are the IDP's killer feature. Without an IDP, creating a new microservice takes two to three weeks: request a repository, configure CI/CD, set up monitoring, create documentation, request cloud resources, configure security scanning, get everything reviewed by platform team. With an AI-powered IDP golden path: developer selects "Python API Service" template, clicks create, receives a fully configured repository with CI/CD pipeline, Prometheus metrics, structured logging, security scanning, API documentation, and a development environment — in forty-five minutes. The golden path encodes every best practice your platform team has learned. Developers can't accidentally skip security scanning because it's baked into the template.

2:00-2:30 - Backstage + AI: The 2026 Stack
Visual: Technology stack layers: Backstage (portal) → AI Agent (automation) → Kubernetes (runtime) → Terraform (infrastructure) → GitOps (deployment).
Voiceover: The dominant IDP technology stack in 2026 centers on Backstage — Spotify's open-source developer portal — enhanced with AI agents. Backstage provides the portal UI: service catalog, documentation, software templates, and plugin ecosystem. AI agents provide the automation layer: auto-populating the catalog, generating templates from existing services, keeping documentation current, and handling provisioning requests. Kubernetes provides the runtime. Terraform provides infrastructure. GitOps provides deployment. This stack is modular — you can adopt pieces incrementally. Start with Backstage service catalog and AI auto-population. Add golden path templates next. Self-service provisioning last.

Before: ticket handlers responding to provisioning requests, manual pipeline configuration, documentation writers whose docs go stale immediately. 3:30-4:00 - Common IDP Mistakes
Visual: Four mistake cards: 1. Building too much too fast, 2. Ignoring developer feedback, 3. No golden paths, 4. Treating it as a tool not a product.
Voiceover: Four mistakes that kill IDP adoption. Building too much too fast: start with service catalog and one golden path template, not a full platform. Ignoring developer feedback: if developers don't want to use it, you've built the wrong thing — interview ten developers before building anything. No golden paths: a catalog without templates is just a wiki — developers still do everything manually. Treating the IDP as a tool instead of a product: assign a product owner, define a roadmap, measure adoption, iterate based on data. The IDPs that fail are always built by platform engineers for platform engineers without input from the developers they're supposed to serve.

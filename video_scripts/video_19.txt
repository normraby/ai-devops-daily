DESCRIPTION: This hands-on tutorial walks through building a functional AI DevOps agent from scratch — from defining capabilities and connecting APIs to deploying your first automated remediation playbook in under two hours.

0:00-0:30 - What We're Building Today
Visual: Final agent architecture preview: Python agent connecting to Kubernetes API, Prometheus, and Slack. Capabilities: monitor, analyze, remediate, notify.
Voiceover: Today we're building a real AI DevOps agent from scratch — not a theoretical architecture, but working code you can deploy this afternoon. Our agent will connect to your Kubernetes cluster, pull metrics from Prometheus, detect anomalies, execute a remediation playbook, and notify your team in Slack. By the end of this tutorial, you'll have a functional agent that monitors pod health, detects crash loops, automatically restarts failing deployments, and reports actions taken. Total build time: under two hours. Prerequisites: Python three dot eleven, kubectl access to a cluster, and a Prometheus instance. Let's build.

0:30-1:00 - Step 1: Project Setup and Architecture
Visual: Terminal showing project creation. File structure: agent.py, config.yaml, playbooks/, connectors/, requirements.txt.
Voiceover: Step one: project setup. Create a new Python project with this structure. Agent dot py is the main orchestration loop. Config dot yaml defines your cluster connection, Prometheus endpoint, Slack webhook, and agent policies. Playbooks directory contains YAML-defined remediation procedures. Connectors directory has modules for Kubernetes, Prometheus, and Slack APIs. Requirements dot txt includes kubernetes, prometheus-api-client, pyyaml, requests, and numpy for anomaly detection. The architecture follows a simple observe-decide-act loop. Every sixty seconds, the agent pulls metrics, compares against baselines, evaluates whether action is needed, executes the appropriate playbook, and logs the result. This is the same pattern every production AI DevOps agent uses — just scaled up.

1:00-1:30 - Step 2: Kubernetes Connector
Visual: Code walkthrough of kubernetes connector module. Functions: get_pods, get_deployments, restart_deployment, get_pod_logs.
Voiceover: Step two: build the Kubernetes connector. This module wraps the Kubernetes Python client with four functions your agent needs. Get pods: returns all pods with their status, restart count, and resource usage. Get deployments: returns deployment status including desired versus ready replicas. Restart deployment: triggers a rolling restart by updating the deployment annotation — the standard Kubernetes restart pattern. Get pod logs: retrieves recent logs from a failing pod for diagnostic context. Load kubeconfig from the default location or in-cluster service account. Test the connector by listing pods in your staging namespace. If you see pod data, your connector works. This module is reusable across every agent you build — invest time here.

1:30-2:00 - Step 3: Prometheus Metrics Connector
Visual: Code walkthrough of Prometheus connector. Queries: pod_cpu_usage, pod_memory_usage, pod_restart_rate, request_error_rate.
Voiceover: Step three: the Prometheus connector. Connect to your Prometheus server and define four queries your agent needs. Pod CPU usage: rate of CPU consumption over five-minute windows. Pod memory usage: current memory working set bytes. Pod restart rate: increase in restart count over fifteen minutes — the key crash loop indicator. Request error rate: percentage of five-xx responses over five-minute windows. The connector returns structured data the agent can evaluate. Store query results with timestamps for trend analysis. The agent needs at least three data points to detect a trend, which is why the observe loop runs every sixty seconds and maintains a rolling window of the last thirty measurements per metric per pod.

2:00-2:30 - Step 4: Anomaly Detection Engine
Visual: Anomaly detection logic flowchart. Baseline calculation → Current value comparison → Threshold evaluation → Severity classification.
Voiceover: Step four: the anomaly detection engine. This is the agent's brain. For each metric, maintain a rolling baseline — the mean and standard deviation of the last thirty measurements. A value exceeding two standard deviations from the mean triggers a warning. Three standard deviations triggers a critical alert. For restart rate specifically, any pod with more than three restarts in fifteen minutes is classified as crash looping — that's a critical alert regardless of statistical deviation. Severity classification drives playbook selection: warnings are logged and monitored, criticals trigger automated remediation. The detection engine is intentionally simple for this tutorial — production agents use more sophisticated models, but statistical baselines catch eighty percent of common issues and are easy to understand and debug.

2:30-3:00 - Step 5: Remediation Playbooks
Visual: YAML playbook file for crash loop restart. Pre-conditions, steps, verification, rollback sections clearly labeled.
Voiceover: Step five: define remediation playbooks in YAML. Here's our crash loop restart playbook. Name: restart crash looping deployment. Pre-conditions: pod restart count exceeds three in fifteen minutes, deployment has more than zero ready replicas — meaning it's not a deployment issue but a runtime issue. Steps: one, collect pod logs for diagnostics. Two, restart the deployment via rolling restart. Three, wait sixty seconds. Four, verify restart count is zero and ready replicas match desired. Rollback: if verification fails after two attempts, escalate to Slack with full diagnostic context and stop auto-remediation. This playbook structure — pre-conditions, steps, verification, rollback — is the standard pattern for every agent playbook. Make playbooks readable, testable, and idempotent.

3:00-3:30 - Step 6: The Agent Loop
Visual: Main agent loop code walkthrough. while True: observe, analyze, decide, act, sleep(60). Logging at each step.
Voiceover: Step six: wire everything together in the main agent loop. Every sixty seconds: observe — pull metrics from Prometheus and pod status from Kubernetes. Analyze — run anomaly detection on all metrics for all pods. Decide — for each critical anomaly, check if a matching playbook exists and pre-conditions are met. Act — execute the playbook steps in order, logging each action. Notify — send Slack message with action taken, diagnostic data, and verification result. Sleep sixty seconds and repeat. The loop is deliberately simple. Production agents add: multiple playbook matching, concurrent execution, rate limiting, and state persistence. But this loop is functional and deployable. Run it against your staging cluster and watch it detect and remediate a simulated crash loop.

3:30-4:00 - Step 7: Slack Integration and Notifications
Visual: Slack notification examples: Warning alert (yellow), Critical with auto-remediation (green), Escalation requiring human (red).
Voiceover: Step seven: Slack notifications. Three notification types. Warning: yellow attachment with metric name, current value, baseline, and deviation. No action taken — monitoring only. Critical with auto-remediation: green attachment showing the anomaly detected, playbook executed, steps taken, and verification result. Escalation: red attachment when auto-remediation failed, including pod logs, metric history, attempted actions, and recommended manual steps. Format notifications for mobile — on-call engineers read these on phones at three AM. Include direct links to Grafana dashboards and Kubernetes resources. Good notifications are the difference between an agent your team trusts and one they disable because it spammed the channel.

4:00-4:30 - Step 8: Deploy and Test
Visual: Deployment checklist: Deploy to staging, Inject crash loop failure, Verify agent detection and remediation, Check Slack notification, Review logs.
Voiceover: Step eight: deploy and test. Deploy the agent as a Kubernetes deployment in your staging cluster — it should run in the same cluster it monitors, using in-cluster service account authentication. Inject a test failure: deploy a pod with a command that exits immediately, creating a crash loop. Within three minutes, your agent should detect the crash loop, execute the restart playbook, and post results to Slack. If it doesn't work, check agent logs first — the observe-decide-act loop logs every step. Common issues: RBAC permissions missing for pod restart, Prometheus query returning empty results, Slack webhook URL incorrect. Fix, redeploy, retest. Once staging works, add production guardrails — require human approval for production namespace actions — and deploy.
